# Coding Constraints - 编码约束规范

> **TL;DR**: 遵循编码约束，避免常见问题。

---

## 1. 语言约束 — 防「中英混杂」

| 约束 | 说明 |
|------|------|
| **输出语言与用户输入一致** | 中文输入→中文输出，英文输入→英文输出 |
| **技术术语保留英文** | API、CRUD、JWT、SQL 等保持英文 |
| **模糊时默认中文** | 语言不明确时默认中文 |

### 示例

```markdown
# ✅ 正确
用户输入中文 → 输出中文
用户输入英文 → 输出英文
技术术语 → 保持英文 (API, JWT, SQL)

# ❌ 错误
中英混杂：这是 a very important API call
```

## 2. 代码审查约束 — 防「漏检和误报」

| 约束 | 说明 |
|------|------|
| **Confidence Score** | 每个问题评 0-100 分 |
| **只报告 ≥ 80 分的问题** | 过滤低置信度问题，减少噪音 |
| **分级显示** | Critical (90-100)、Important (80-89) |
| **质量优于数量** | 严格过滤，只报告真正重要的问题 |

### Confidence Score 标准

| 分数 | 级别 | 说明 |
|------|------|------|
| 90-100 | Critical | 必须修复，阻止合并 |
| 80-89 | Important | 建议修复，需要讨论 |
| < 80 | - | 不报告（噪音） |

## 3. 调试约束 — 防「修完就跑」

### 调试步骤

| 步骤 | 要求 |
|------|------|
| 1. 捕获完整错误信息 | 错误消息、堆栈跟踪、环境详情 |
| 2. 查 git diff | 检查引入问题的最近变更 |
| 3. 最小复现步骤 | 定位精确的失败位置 |
| 4. 二分定位 | 通过注释代码段隔离问题区域 |
| 5. 定向修复 | 最小化副作用的精确修复 |
| 6. 验证 | 确认修复有效且不破坏现有功能 |

### 交付物（每次调试必须提供）

| 交付物 | 说明 |
|--------|------|
| **Root Cause** | 清晰解释问题发生的原因 |
| **Evidence** | 证明诊断的具体代码/日志 |
| **Fix** | 解决问题的最小代码变更 |
| **Verification** | 确认修复的测试用例或命令 |
| **Prevention** | 避免类似问题的建议 |

## 4. 实现约束 — 防「实现跑偏」

### DO（必须做）

| 必须做 | 说明 |
|--------|------|
| ✅ 严格按照架构规格执行 | 不偏离决策 |
| ✅ 实现 PRD 中所有验收标准 | 完成所有验收标准 |
| ✅ 为所有业务逻辑编写测试 | 测试覆盖 |
| ✅ 包含完整的错误处理 | try/catch, error handling |
| ✅ 添加适当的日志记录 | logging |
| ✅ 遵循安全最佳实践 | security |
| ✅ 使用环境变量管理配置 | env vars |
| ✅ 实现正确的数据校验 | validation |
| ✅ 处理边界情况 | edge cases |

### DON'T（禁止做）

| 禁止做 | 说明 |
|--------|------|
| ❌ 偏离架构决策 | - |
| ❌ 跳过错误处理 | - |
| ❌ 硬编码敏感信息 | - |
| ❌ 忽略安全考虑 | - |
| ❌ 写未经测试的代码 | - |
| ❌ 创建过于复杂的解决方案 | - |
| ❌ 在单个函数中混合多个关注点 | - |
| ❌ 跳过输入校验 | - |

## 5. 代码质量原则

| 原则 | 说明 |
|------|------|
| **KISS** | Keep It Simple, Stupid — 简单优先 |
| **DRY** | Don't Repeat Yourself — 避免重复 |
| **YAGNI** | You Aren't Gonna Need It — 不提前实现不需要的功能 |
| **SOLID** | 单一职责、开闭原则、里氏替换、接口隔离、依赖倒置 |

---

## 6. 禁止模式

| 禁止 | 替代方案 |
|------|---------|
| ❌ 嵌套三元运算符 | 用 switch 或 if/else 链替代 |
| ❌ 行数优先于可读性 | 选清晰而非简洁 |
| ❌ 过度抽象 | 避免 combine too many concerns into single functions |
| ❌ 魔法数字/字符串 | 用常量替代 |

---

## 7. 文件长度约束

| 约束 | 阈值 | 处理方式 |
|------|------|---------|
| **单文件最大行数** | 300 行 | 超过则拆分 |
| **拆分原则** | - | 按业务/功能模块分离 |

---

## 8. 代码审查清单

### 提交代码前，AI 必须检查：

| 检查项 | 问题 |
|--------|------|
| 函数是否超过 40 行？ | 需要拆分 |
| 嵌套是否超过 3 层？ | 需要提取函数 |
| 函数是否只做一件事？ | 违反 SRP |
| 是否有魔法数字/字符串？ | 需要替换为常量 |
| 是否有重复代码？ | 需要抽象 |
| 文件是否超过 300 行？ | 需要拆分 |

### 函数复杂度标准

| 标准 | 阈值 |
|------|------|
| 函数最大行数 | 40 行 |
| 最大嵌套层数 | 3 层 |
| 最大参数个数 | 4 个 |
| 文件最大行数 | 300 行 |
